# Website Content Extractor — 项目说明与使用指南

本文档面向不熟悉项目结构和术语的读者，用中文通俗、完整地说明本仓库的作用、用法和后续可改进方向。

---

## 一、这个项目是做什么的？

**一句话：** 这是一个「网页内容抓取器」—— 你给它一个或多个网址，它按你的设置去访问这些页面（以及从页面上发现的更多链接），把每个页面的**正文内容、标题、描述、内外链数量**等整理成一条条结构化数据，方便你导出、分析或做后续处理。

**典型用途：**
- 给网站做内容盘点、SEO 或质量审计
- 把一批页面的文字抓下来做检索、归档或训练数据
- 先快速发现站内所有链接，再决定抓哪些
- 监控某些页面内容是否变化

**技术身份：** 本项目是一个 **Apify Actor**，即可以在 Apify 平台上以「输入 → 运行 → 得到数据集」的方式使用的自动化任务；同时也可以在你自己电脑或服务器上本地运行、或打成 Docker 镜像运行。

## 二、核心概念与术语（通俗解释）

| 术语 | 通俗解释 |
|------|----------|
| **Apify** | 一个云端的自动化/爬虫平台。你可以在网页上创建任务、填参数、点运行，结果会存成「数据集」并可以导出。 |
| **Actor** | Apify 里的一类可运行程序。每个 Actor 有固定「输入」和「输出」约定，本仓库就是这样一个 Actor 的完整代码。 |
| **输入 (Input)** | 你运行前填写的参数，例如：从哪些网址开始、最多抓多少页、是否只抓同域名、要不要等某个元素出现再抓等。所有可填项在 `.actor/input_schema.json` 里定义。 |
| **输出 (Output)** | 运行结束后产生的结果。主要有两部分：**数据集 (Dataset)** 里每条记录是一页的抓取结果；**Key-Value 存储**里会存一份「运行摘要 (runSummary)」，包含总页数、成功/失败数、失败 URL 列表等。 |
| **Dataset（数据集）** | 表状数据：每一行对应一个被访问的 URL，列包括 url、success、content、title、content_length、links_internal_count 等。可在 Apify 控制台里查看、导出 CSV/JSON。 |
| **runSummary（运行摘要）** | 一次运行的高层统计与失败列表：例如 totalPages、successCount、failedCount、errorTypes、totalContentLength、**failedUrls**（失败 URL 列表，便于重试）。 |
| **crawl4ai** | 本项目依赖的一个开源库，负责「用浏览器打开网页、等加载、抽正文、抽链接」等底层工作。本 Actor 是在其之上做了配置、封装和 Apify 对接。 |
| **startUrls（起始 URL）** | 必填。程序从这些网址开始访问；在「全量抓取」模式下还会根据设置继续跟踪页面上的站内链接。 |
| **深度 (maxDepth)** | 从起始 URL 算起，最多跟几层链接。0 = 只抓起始页；1 = 起始页 + 起始页上的链接；2 = 再多一层，以此类推。 |
| **full / discover_only 模式** | **full**：正常抓取，每条结果带正文、摘要等。**discover_only**：只拿 URL、状态、标题和内外链列表，不抓正文，适合「先发现链接」或做站点地图。 |

## 三、项目目录结构（每个是干什么的）

```
项目根目录/
├── .actor/                    # Apify Actor 的配置与 schema（平台用）
│   ├── actor.json             # Actor 元信息：名称、描述、输入/输出/存储指向
│   ├── input_schema.json       # 输入参数的完整定义（类型、默认值、说明）
│   ├── output_schema.json      # 输出结构的 schema
│   └── dataset_schema.json     # 数据集每条记录的字段定义与视图
├── src/
│   └── crawl4ai_actor/         # 主要代码包
│       ├── main.py             # 入口：读输入、调爬虫、写结果、写 runSummary
│       ├── config.py           # 用 Pydantic 定义并校验「输入」模型
│       ├── crawler.py          # 核心爬虫逻辑：调度 crawl4ai、解析结果、按模式输出
│       └── __init__.py
├── tests/                     # 单元测试
│   ├── test_config.py         # 输入校验、别名、默认值
│   ├── test_crawler_utils.py  # URL 规范化、等待选择器、错误分类等工具函数
│   └── test_main.py           # runSummary 结构等
├── scripts/
│   └── ux_matrix.py          # 回归脚本：用多组真实场景（不同网站/配置）跑一遍，检查是否正常
├── Dockerfile                 # 构建可在 Apify 或本地跑的镜像
├── pyproject.toml             # 项目依赖与 Python 配置（含 pytest、ruff）
├── README.md                  # 英文简短说明与快速开始
└── 项目说明与使用指南.md       # 本文件：中文详细说明
```

- **改输入/输出约定**：动 `.actor/*.json` 和 `config.py`。
- **改抓取行为**：主要改 `src/crawl4ai_actor/crawler.py`。
- **改入口与摘要、失败列表**：改 `main.py`。

## 四、怎么用（三种方式）

### 1. 在 Apify 平台上用

1. 把本仓库部署为 Apify Actor（或使用已部署好的）。
2. 在 Actor 页面填写「输入」：至少填 **startUrls**（起始网址列表），其余用默认即可。
3. 点运行。跑完后在「Storage」里看 **Dataset**（每条记录一页）和 **Key-value store** 里的 **runSummary**（含 failedUrls 等）。
4. 需要重试失败页时，把 runSummary 里的 **failedUrls** 复制出来，作为下一次运行的 **startUrls** 再跑一遍即可。

### 2. 在本地用（命令行）

```bash
# 安装依赖并安装本包
pip install -e ".[dev]"
# 首次使用需执行 crawl4ai 的浏览器环境安装
crawl4ai-setup

# 运行（会从 Apify 默认输入源读参数；本地调试时可改代码里默认输入或通过环境变量等传入）
python -m crawl4ai_actor.main
```

本地运行时，若没有 Apify 运行时环境，需要自己在代码或脚本里构造输入字典并传给逻辑（或写一个本地小脚本读 JSON 再调 `crawl_urls`）。

### 3. 用 Docker 跑

```bash
docker build -t website-content-extractor .
# 之后通过 Apify 的 Docker 运行方式或自行挂载输入/输出卷运行
```

适合在服务器或 Apify 的 Docker 环境中一致地运行同一套环境。

## 五、输入参数说明（完整一览）

- **必填**
  - **startUrls**（数组）：至少一个起始 URL，如 `["https://example.com"]`。

- **常用**
  - **maxPages**：最多抓多少页（默认 50）。
  - **maxDepth**：链接深度（默认 2）。
  - **extractMode**：正文格式 —— `markdown` / `html` / `text`（默认 markdown）。
  - **crawlMode**：`full`（默认，正常抓正文）或 `discover_only`（只拿 URL + 链接列表，不抓正文）。
  - **sameDomainOnly**：是否只跟同域名链接（默认 true）。
  - **includePatterns / excludePatterns**：用正则过滤要抓或要排除的 URL。

- **与「等待/慢站/SPA」相关**
  - **waitUntil**：页面加载策略 —— `domcontentloaded`（快）/ `load` / `networkidle`（适合 SPA 或慢站）。
  - **pageLoadWaitSecs**：加载后再额外等几秒再抓（小数也可）。
  - **waitForSelector**：等某个 CSS 选择器出现再抓（可加前缀 `css:` 或 `js:`）。
  - **waitForTimeoutSecs**：等选择器的最长时间（秒）。

- **与「只抓某一块」相关**
  - **cssSelector**：只抽取页面中该选择器内的内容（如 `main`、`.article`）。

- **与「链接输出」相关**
  - **includeLinkUrls**：为 true 时，每条结果里带 **links_internal**、**links_external** 数组（默认 false，只有数量）。

- **其它常用**
  - **concurrency**：并发数。
  - **requestTimeoutSecs**：单页超时（秒）。
  - **maxRetries** / **retryBackoffSecs**：失败重试次数与退避。
  - **maxRequestsPerMinute**：全局限速（0 表示不限制）。
  - **virtualScrollSelector** / **virtualScrollCount**：针对无限滚动页面，先滚动再抓。
  - **cleanContent**：是否做导航/噪音清理（默认 true）。
  - **maxResults**：最多往数据集里写多少条（默认 1000）。

完整列表与默认值、取值范围以 **.actor/input_schema.json** 为准。

## 六、输出说明（每条记录 + runSummary）

### 6.1 数据集里每条记录（full 模式）

| 字段 | 含义 |
|------|------|
| url | 当前页 URL |
| success | 是否抓取成功 |
| status_code | HTTP 状态码（若有） |
| error_message / error_type | 失败时的错误信息与分类（如 rate_limited、blocked、network_error） |
| content | 正文（经清理、截断后，依 extractMode 为 markdown/html/text） |
| content_raw | 未清理的原始正文（仅当 includeRawContent 为 true 时存在） |
| content_excerpt | 前 N 字的摘要（N 由 contentExcerptChars 控制） |
| content_truncated | 是否被 maxContentChars 截断 |
| title / meta_description | 页面标题与 meta 描述 |
| content_length / content_hash | 正文长度与哈希（便于查重） |
| links_internal_count / links_external_count | 内外链数量 |
| links_internal / links_external | 内外链 URL 数组（仅当 includeLinkUrls 为 true 时存在） |
| extracted_at | 抓取时间（ISO） |
| retry_attempt / will_retry | 当前重试次数、是否还会重试 |

**discover_only 模式**下，每条记录只有：url、success、status_code、error_type、title、links_internal、links_external、*_count、extracted_at，没有 content 等正文字段。

### 6.2 运行摘要 runSummary（Key-Value 存储）

- **totalPages**：本次处理的总页数。
- **successCount** / **failedCount**：成功/失败数。
- **errorTypes**：按错误类型统计的数量（如 page_error、network_error）。
- **totalContentLength**：成功页正文总字符数。
- **failedUrls**：所有失败页的 URL 列表，便于单独重试或排查。

## 七、两种运行模式与典型场景

- **full（默认）**  
  - 适合：要正文、要做内容分析、归档、检索。  
  - 可配合 waitUntil / pageLoadWaitSecs / waitForSelector / cssSelector 应对 SPA 或只要某一块区域。

- **discover_only**  
  - 适合：先摸清站内有哪些链接、做站点地图、或为下一轮「只抓部分 URL」生成 startUrls。  
  - 输出更小、更快，不抽正文。

**示例（仅发现链接）：**

```json
{
  "startUrls": ["https://example.com"],
  "crawlMode": "discover_only",
  "maxPages": 100
}
```

**示例（SPA/慢站）：**

```json
{
  "startUrls": ["https://example.com"],
  "waitUntil": "networkidle",
  "pageLoadWaitSecs": 2
}
```

## 八、后续可改进方向（想法汇总）

- **LLM 抽取**：接入大模型做正文摘要、关键信息抽取、分类等，需配置 API Key，适合作为「高级模式」或单独迭代。
- **Prefetch / 仅发现**：当前 discover_only 已覆盖「只拿链接」；若需「先发现再按规则分批抓」，可在现有 failedUrls/startUrls 流程上做调度或拆成多轮运行。
- **Deep crawl 策略**：当前是自管的 BFS 式 frontier；若要与 crawl4ai 官方深度策略对齐，可再抽象一层策略接口。
- **更多输出格式**：如导出为 CSV 时的列选择、或按域名/路径分文件等，可在 Actor 或后处理脚本里加。
- **可访问性/SEO 元数据**：在输出中增加如 h1、图片 alt、结构化数据等，需在 crawler 层从 crawl4ai 结果里多取字段。

以上均为在「保证功能、极简仓库」前提下的可选增强，按需排期即可。

## 十、常见问题与注意点

1. **输入报错「startUrls: field required」**  
   表示没填必填项 startUrls，或填成了空数组。必须至少一个合法 http(s) URL。

2. **结果里 content 为空**  
   可能是页面需要登录、被反爬、或加载太慢。可尝试：调大 requestTimeoutSecs、设 waitUntil 为 networkidle、加 pageLoadWaitSecs、或设 waitForSelector 等页面主体出现。

3. **想只抓某一块区域**  
   使用 **cssSelector**（如 `main`、`.article-body`）即可只抽该区域内容。

4. **失败了很多页，想重试**  
   在 Storage 的 Key-Value 里打开 runSummary，复制 **failedUrls**，作为下一次运行的 **startUrls** 再跑一次即可。

5. **正则不会写**  
   includePatterns/excludePatterns 可以留空；只靠 sameDomainOnly 和 maxDepth 也能用。需要时再查正则语法或先用简单字符串匹配再逐步改成正则。

6. **和 crawl4ai 的关系**  
   本仓库是「用 crawl4ai 做抓取引擎、自己管输入输出与 Apify 对接」的 Actor；crawl4ai 的版本与能力会直接影响抓取效果，大版本升级时需跑一遍 ux_matrix 做回归。
